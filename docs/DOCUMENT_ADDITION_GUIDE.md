## 1. Обзор системы

### Архитектура хранения

```
Документ (JSON)
      ↓
  Chunking Service
      ↓
  Chunks (части документа)
      ↓
  IndexedDB Storage
      ↓
  Hybrid Search (Keyword 40% + Vector 60%)
      ↓
  Результаты поиска → LLM → Ответ пользователю
```

### Ключевые компоненты

| Компонент | Файл | Назначение |
|-----------|------|------------|
| Хранение | `services/indexedDBService.ts` | IndexedDB для документов и chunks |
| Chunking | `services/chunkingService.ts` | Разбиение документов на части |
| Поиск | `services/searchService.ts` | Гибридный поиск (keyword + vector) |
| Инициализация | `hooks/useDatabase.ts` | Загрузка и индексация при старте |

### Расположение файлов

```
/public/data/
├── documents/
│   ├── TRUDOVOI_KODEKS.json      # Документы
│   ├── DEKRET_8.json
│   ├── SOTRUDNIKI_2025.json
│   ├── KLIENTY_2025.json
│   └── ... (другие документы)
└── documents-index.json           # Индекс всех документов
```

## 2. Форматы документов

### 2.1 Обязательные поля

Каждый документ **обязательно** должен содержать:

```json
{
  "id": "UNIQUE_ID",           // Уникальный ID (UPPERCASE, без пробелов)
  "title": "Полное название",  // Отображается в UI
  "type": "DOC",               // Тип: DOC, PDF, LAW
  "citation": "Краткое имя",   // Используется в цитатах ответов
  "fullContent": "Текст..."    // ОБЯЗАТЕЛЬНО! Полный текст документа
}
```

### 2.2 Опциональные поля

```json
{
  "url": "https://...",              // Ссылка на источник
  "adoptedDate": "2025-01-01",       // Дата принятия
  "lastUpdated": "2025-12-18T...",   // Дата обновления
  "sections": [...]                   // Структура документа (см. ниже)
}
```

## 3. Процесс индексации

### 3.1 Автоматическая индексация при запуске

При запуске приложения (`npm run dev`) происходит:

```
┌─────────────────────────────────────────────────────────────┐
│  1. Инициализация IndexedDB                                  │
│     └─ Создание stores: documents, chunks, chats, settings   │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  2. Проверка миграции (localStorage → IndexedDB)             │
│     └─ Если есть старые данные - мигрируем                   │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  3. Загрузка документов из IndexedDB                         │
│     └─ Если < 15 документов - импорт из /public/data/        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  4. Проверка chunks                                          │
│     └─ Найти документы без chunks                            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  5. Генерация chunks для новых документов                    │
│     └─ chunkingService.chunkDocument(doc) для каждого        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  6. Batch-сохранение всех chunks                             │
│     └─ indexedDB.saveChunks(allNewChunks) - одна транзакция  │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  7. База готова к поиску                                     │
└─────────────────────────────────────────────────────────────┘
```

## 4. Chunking - разбиение на части

### 4.1 Зачем нужен chunking?

- Документы могут быть **150+ KB** текста
- LLM имеет ограниченный контекст
- Поиск точнее по **релевантным частям**, а не по всему документу

### 4.2 Логика chunking (chunkingService.ts)

```
                    Документ
                        │
        ┌───────────────┼───────────────┐
        ↓               ↓               ↓
   Есть sections    Есть "Статья N"   Простой текст
   с content?       в fullContent?
        │               │               │
        ↓               ↓               ↓
   По статьям       Парсинг по       По строкам или
   из структуры     паттерну         весь документ
```

### 4.3 Типы chunks

| Тип | Описание | Пример path |
|-----|----------|-------------|
| `article` | Статья закона | `РАЗДЕЛ I → ГЛАВА 1 → Статья 28` |
| `chapter` | Глава без статей | `РАЗДЕЛ I → ГЛАВА 1` |
| `section` | Раздел без глав | `РАЗДЕЛ I` |

### 4.4 Особые случаи

**Документы "Сотрудники" / "Клиенты":**
- Определяются по `citation` содержащему "сотрудник" или "клиент"
- Используют упрощённый path: `пункт 1`, `пункт 2`
- Получают **POST-FUSION bonus** в поиске (+100 к score)

### 4.5 Структура chunk

```typescript
interface Chunk {
  id: string;           // "DOC_ID_S1_C1_A28"
  sourceId: string;     // ID исходного документа
  sourceTitle: string;  // Название документа
  citation: string;     // Краткое имя для цитирования
  path: string;         // Путь: "РАЗДЕЛ I → ГЛАВА 1 → Статья 28"
  content: string;      // Текст chunk'а
  chunkType: string;    // 'article' | 'chapter' | 'section'
}
```

```bash
# Создайте файл в папке documents
touch /public/data/documents/MY_NEW_DOC.json
```

```json
{
  "id": "MY_NEW_DOC",
  "title": "Мой новый документ",
  "type": "DOC",
  "citation": "Новый документ",
  "fullContent": "Полный текст документа здесь..."

```

Откройте `/public/data/documents-index.json` и добавьте ID:

```json
{
  "documents": [
    "TRUDOVOI_KODEKS",
    "DEKRET_8",
    "SOTRUDNIKI_2025",
    "KLIENTY_2025",
    "MY_NEW_DOC"
  ]
}
```

### Шаг 4: Перезапустите приложение

```bash
# Остановите текущий сервер (Ctrl+C) и запустите заново
npm run dev
```

### Шаг 5: Проверьте индексацию

Откройте консоль браузера (F12 → Console) и найдите:

```
✅ Imported 1 new documents
📝 MY_NEW_DOC: создано X chunks
```

## 7. Настройка поиска

### 7.1 Гибридный поиск

Система использует комбинацию двух методов:

```
┌─────────────────┐     ┌─────────────────┐
│  Keyword Search │     │  Vector Search  │
│     (40%)       │     │     (60%)       │
└────────┬────────┘     └────────┬────────┘
         │                       │
         └───────────┬───────────┘
                     ↓
              ┌──────────────┐
              │     RRF      │
              │ (Reciprocal  │
              │ Rank Fusion) │
              └──────┬───────┘
                     ↓
              ┌──────────────┐
              │ POST-FUSION  │
              │   Bonuses    │
              └──────┬───────┘
                     ↓
              Финальный рейтинг
```

### 7.2 POST-FUSION бонусы

Для специальных типов документов применяются бонусы **после** RRF:

| Тип документа | Условие | Бонус |
|---------------|---------|-------|
| Сотрудники | citation содержит "сотрудник" | +100 |
| Клиенты | citation содержит "клиент" | +100 |
| Точная дата | Совпадение "25 декабря" | +150 |

### 7.3 Добавление бонуса для нового типа

Если нужно, чтобы новый тип документа (например, "Продукты") получал приоритет, добавьте в `searchService.ts`:

```typescript
// В методе hybridSearchChunks(), после RRF (примерно строка 400+)

// Определяем тип запроса
const isProductQuery = /продукт|товар|решение|система|цена/i.test(lowerQuery);

// В цикле combinedScores.forEach():
if (lowerCitation.includes('продукт') && isProductQuery) {
  entry.score += 100; // POST-FUSION bonus для продуктов
}
```

### 7.4 Ключевые слова по отделам

В `searchService.ts` есть словарь `departmentKeywords`:

```typescript
const departmentKeywords: Record<string, string[]> = {
  hr: ['отпуск', 'зарплата', 'увольнение', 'трудовой', 'сотрудник', ...],
  legal: ['договор', 'суд', 'иск', 'закон', ...],
  accounting: ['налог', 'бухгалтер', 'счет', 'НДС', ...],
  it: ['программ', 'система', 'безопасность', ...]
};
```

Добавьте релевантные ключевые слова для улучшения поиска.

## 8. Проверка и отладка

### 8.1 Проверка в консоли браузера

```javascript
// Открыть DevTools (F12) → Console

// Проверить количество документов
const docs = await indexedDB.getAllDocuments();
console.log('Документов:', docs.length);

// Проверить chunks
const chunks = await indexedDB.getAllChunks();
console.log('Chunks:', chunks.length);

// Найти chunks для конкретного документа
const myChunks = chunks.filter(c => c.sourceId === 'MY_NEW_DOC');
console.log('Chunks для MY_NEW_DOC:', myChunks);
```

### 8.2 Проверка в IndexedDB

1. Откройте DevTools (F12)
2. Перейдите в Application → IndexedDB → BelhardAI_DB
3. Проверьте stores:
   - `documents` - все документы
   - `chunks` - все chunks

### 8.3 Тестирование поиска

После добавления документа протестируйте поиск:

1. Откройте приложение
2. Введите запрос, связанный с новым документом
3. Проверьте, что документ появляется в источниках ответа

### 8.4 Debug-режим chunking

```javascript
// В консоли браузера
import { chunkingService } from './services/chunkingService';

// Получить документ
const doc = await indexedDB.getDocument('MY_NEW_DOC');

// Отладка chunking
chunkingService.debugDocumentChunks(doc);
```

## 9. Частые ошибки

### 9.1 Документ не индексируется

**Причина:** Не добавлен в `documents-index.json`

**Решение:**
```json
// /public/data/documents-index.json
{
  "documents": [
    ...,
    "MY_NEW_DOC"  // Добавить ID
  ]
}
```

### 9.2 Chunks не создаются

**Причина:** Пустой `fullContent` или `content` в articles

**Решение:** Убедитесь, что:
- `fullContent` содержит полный текст
- Или `articles[].content` заполнен

### 9.3 Документ не находится в поиске

**Причины:**
1. Chunks не созданы (см. выше)
2. Нет релевантных ключевых слов
3. Слишком короткий контент

**Решение:**
- Проверьте логи: `📝 MY_NEW_DOC: создано X chunks`
- Добавьте больше текста в `fullContent`
- Используйте ключевые слова в тексте

### 9.4 Неправильное цитирование в ответах

**Причина:** Неправильный формат `citation`

**Решение:**
```json
{
  "citation": "Краткое имя"  // Это будет в цитатах
}
```

### 9.5 Ошибка парсинга JSON

**Причина:** Синтаксическая ошибка в JSON

**Решение:**
- Проверьте JSON на https://jsonlint.com/
- Убедитесь, что все кавычки экранированы: `\"текст\"`
- Проверьте запятые между элементами
